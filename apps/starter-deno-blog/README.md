# starter-deno-blog

Minimal blog built with [Deno 2](https://deno.com) and LaikaCMS. Demonstrates the zero-adapter
integration pattern:

- **`Deno.serve()` is WHATWG-native** ‚Äî the fetch handler receives and returns `Request`/`Response`
  directly. `laika.fetch(request)` is the entire Decap proxy ‚Äî no `IncomingMessage` bridge needed.
- **`createEmbeddedLaika` works on Deno** ‚Äî it uses `node:fs` and `node:path` internally; Deno's
  Node.js compatibility layer handles both. Requires `--allow-read --allow-write` permissions.
- **`import.meta.dirname`** ‚Äî Deno 1.28+ and Node 21.2+ expose the directory of the current source
  file, useful for computing absolute paths without relying on `Deno.cwd()` / `process.cwd()`.
- **Decap admin from CDN** ‚Äî `admin-client.ts` bundled by esbuild, same pattern as all other
  starters.

## Quick start

```bash
pnpm install
pnpm dev         # builds admin bundle then starts Deno dev server with --watch
```

Open <http://localhost:3000> for the blog and <http://localhost:3000/admin/> for the CMS editor (dev
auth ‚Äî no login required).

## Why Deno.serve() needs no bridge

Deno's `Deno.serve()` fetch handler receives a WHATWG `Request`:

```ts
Deno.serve({ port: 3000 }, async (request: Request) => {
  if (url.pathname.startsWith('/api/decap')) {
    // request IS a WHATWG Request ‚Äî pass directly, no conversion
    return laika.fetch(request);
  }
});
```

Compare to Express/Koa/Fastify which receive Node.js `IncomingMessage` and require a manual bridge
to construct a WHATWG `Request`. See [docs/decap-integration.md](../../docs/decap-integration.md)
for the bridge pattern.

## Node.js compat in Deno

`createEmbeddedLaika` imports `existsSync`, `mkdirSync`, `writeFileSync` from `node:fs` and
`dirname`, `resolve` from `node:path`. Deno supports the `node:` protocol via its built-in Node.js
compatibility layer ‚Äî no polyfills or shims required.

The `deno.json` tasks include `--allow-read` and `--allow-write` so Deno grants the file system
access these Node.js APIs need.

## Deno permissions

| Permission      | Why it is needed                                               |
| --------------- | -------------------------------------------------------------- |
| `--allow-net`   | HTTP server (`Deno.serve`) + any outbound fetch inside laika   |
| `--allow-read`  | `node:fs` read ops in `createEmbeddedLaika`; serving `public/` |
| `--allow-write` | `node:fs` write ops ‚Äî Decap CRUD creates/updates content files |

For a tighter production deployment you can scope permissions:
`--allow-read=./content,./public --allow-write=./content --allow-net=:3000`

## Auth modes

| Mode     | When to use                                    |
| -------- | ---------------------------------------------- |
| `dev`    | Local development ‚Äî no credentials required    |
| `custom` | Production ‚Äî provide `authenticateAccessToken` |

## Rendering markdown

The `body` field from Decap CMS is raw markdown. This starter renders it in a `<pre>` block. In
production swap it for a Deno-compatible markdown renderer such as
[`@std/markdown`](https://jsr.io/@std/markdown) or `npm:marked`.
